# GoodCabs: End-to-End Medallion Architecture for Ride Analytics Using Databricks

An end-to-end data engineering pipeline built on **Databricks Lakeflow Declarative Pipelines**, implementing the **Medallion Architecture (Bronze → Silver → Gold)** to transform raw, multi-city ride-hailing trip data into clean, business-ready analytics — powering dashboards and natural language querying via Databricks Genie.

---

## 📌 Project Overview

GoodCabs is a fictional multi-city cab company operating across 11 Indian cities. Raw trip data is generated by an application server and stored in a relational database. This project simulates that real-world data flow — ingesting raw trip and city data from cloud storage (S3), progressively cleaning and validating it, and modeling it into a reporting-ready **star schema** — so that business teams can answer questions like:

- Which city generates the most revenue?
- Are ratings or trip volumes different on weekends vs. weekdays?
- How does business change around national holidays?
- What's the performance breakdown for a specific city?

---

## 🏗️ Architecture

![Architecture Diagram](other_files/architecture.png)

**Data Flow:**

```
Application Server → Relational DB → (simulated) Data Fetch Service → S3
        → Databricks Workspace (Bronze → Silver → Gold, via Lakeflow Declarative Pipelines)
        → Dashboards & Genie (Natural Language Q&A)
```

All tables are governed centrally through **Unity Catalog**, under the `transportation` catalog.

---

## 🥉🥈🥇 Medallion Layers

### Bronze — Raw Ingestion
Raw data is ingested as-is from S3, with no transformation, plus metadata columns (`file_name`, `ingest_datetime`) for traceability.

| File | Description |
|---|---|
| `bronze/city.py` | Batch load of city reference data via materialized view |
| `bronze/trips.py` | Streaming ingestion of trip data using **Auto Loader**, with schema evolution support |

### Silver — Cleaned & Validated
Data is cleaned, standardized, and quality-checked.

| File | Description |
|---|---|
| `silver/calendar.py` | Generated date dimension table with year/month/quarter/weekday/Indian holiday attributes |
| `silver/city.py` | Cleaned city dimension with standardized column names |
| `silver/trips.py` | Data quality expectations (valid dates, valid ratings) + **CDC-based upsert (SCD Type 1)** to prevent duplicate/stale trip records |

### Gold — Business-Ready
Clean fact and dimension tables are joined into a single, reporting-ready **star schema**.

| File | Description |
|---|---|
| `gold/trips_gold.sql` | Central `fact_trips` table — joins trips with city and calendar dimensions |
| `gold/city_views/*.sql` | City-specific filtered views (one per city) for scoped regional reporting |

## 🧠 Key Concepts Demonstrated

- **Medallion Architecture** (Bronze / Silver / Gold layering)
- **Lakeflow Declarative Pipelines** (formerly Delta Live Tables)
- **Auto Loader** for incremental streaming ingestion
- **Schema evolution** handling for changing source files
- **Data Quality Expectations** (`dp.expect`) for automated validation
- **Change Data Capture (CDC)** with `create_auto_cdc_flow`, SCD Type 1 upserts
- **Star Schema modeling** (fact + dimension tables)
- **Unity Catalog** for governance and lineage
- **Databricks Genie** for natural language business querying

## ⚙️ How to Run This Project

1. Clone this repository.
2. In Databricks: **Workspace → Repos → Add Repo**, and connect this GitHub repository.
3. Set up an **External Location** and **Storage Credential** in Unity Catalog pointing to your own S3 bucket.
4. Update `SOURCE_PATH` in `bronze/city.py` and `bronze/trips.py` to point to your own S3 bucket path.
5. Create a new **Lakeflow Declarative Pipeline** in Databricks, pointing to the `2. codes/bronze` and `2. codes/silver` folders as source.
6. Run the pipeline — Bronze, Silver, and Gold tables will be created automatically under the `transportation` catalog.
7. Build a Dashboard or connect Genie on top of `transportation.gold.fact_trips` to start querying.

> **Note:** This repo does not include real production credentials or a live S3 bucket. Replace `SOURCE_PATH` with your own bucket before running.
